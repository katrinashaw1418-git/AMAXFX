You will need to align the exact payload shape to your Fireblocks webhook version, but the integration pattern is stable: receive the transaction status update, find the internal transaction, then settle or fail it. Fireblocks documents transaction status webhooks and expects your endpoint to return HTTP 200, retrying if no response is received. 

app.post("/api/webhooks/fireblocks", express.json(), async (req, res) => {
  try {
    const event = req.body;

    // Adjust these field names to your configured webhook version.
    const data = event?.data || event;
    const providerReference = data?.id || data?.txId;
    const status = data?.status;
    const txHash = data?.txHash || data?.txHashs?.[0] || null;
    const externalTxId = data?.externalTxId ? Number(data.externalTxId) : null;

    let txRecord = null;

    if (externalTxId) {
      const rows = await db.select()
        .from(transactions)
        .where(eq(transactions.id, externalTxId));
      txRecord = rows[0] || null;
    }

    if (!txRecord && providerReference) {
      const rows = await db.select()
        .from(transactions)
        .where(eq(transactions.providerReference, providerReference));
      txRecord = rows[0] || null;
    }

    if (!txRecord) {
      return res.status(200).json({ ok: true });
    }

    if (status === "COMPLETED") {
      const wallet = await getWallet(txRecord.userId, txRecord.fromCurrency);
      if (!wallet) {
        return res.status(200).json({ ok: true });
      }

      await db.transaction(async (tx) => {
        await applyCompletedWithdrawal(tx, wallet, txRecord.amount);
        await markTransactionCompleted(tx, txRecord.id, {
          providerReference,
          blockchainTxHash: txHash,
        });
      });

      return res.status(200).json({ ok: true });
    }

    if (status === "FAILED" || status === "CANCELLED" || status === "REJECTED") {
      const wallet = await getWallet(txRecord.userId, txRecord.fromCurrency);
      if (!wallet) {
        return res.status(200).json({ ok: true });
      }

      const amountPlusFee = new Decimal(txRecord.amount).plus(txRecord.fee || 0);

      await db.transaction(async (tx) => {
        await rollbackFailedWithdrawalReservation(tx, wallet, amountPlusFee);
        await markTransactionFailed(tx, txRecord.id, {
          providerReference,
          failureReason: `Fireblocks status: ${status}`,
        });
      });

      return res.status(200).json({ ok: true });
    }

    return res.status(200).json({ ok: true });
  } catch (error) {
    console.error("Fireblocks webhook processing failed", error);
    return res.status(500).send("Webhook processing failed");
  }
});
Fireblocks also notes that for balance validation and reconciliation, incoming transactions should only be considered reconciled after COMPLETED and the related balance update webhook has been received. That is why your production reconciliation job should not treat a merely detected deposit as settled yet.

app.post("/api/crypto/deposits/address", requireAuth, async (req, res) => {
  try {
    const userId = req.user!.id;
    const { currency } = req.body;

    if (!currency) {
      return res.status(400).json({ message: "Currency is required" });

    const assetId = String(currency).toUpperCase();

14. What this gives you immediately
After these patches, your app becomes much closer to a real settlement architecture:

Fiat card deposits become pending -> Stripe PaymentIntent -> Stripe webhook -> wallet credit, which matches Stripe’s intended post-payment flow. 

Crypto withdrawals become pending -> reserve available balance -> Fireblocks transaction -> Fireblocks webhook -> reduce actual balance / mark failed, which matches Fireblocks’ transaction lifecycle model. 

15. Two important caveats
Stripe solves card deposits, not your Australian bank transfer withdrawal rail. For that you still need a banking/payments provider such as Monoova/Cuscal or another payout rail. I am not claiming Stripe alone makes fiat withdrawal production-ready.

Fireblocks’ exact webhook payload fields can vary by webhook version/configuration, so you will need one sandbox test pass to align the JSON field names in the handler with your workspace configuration. Fireblocks’ docs explicitly reference webhook versioning and migration guidance.
