The Correct Google Sign-In Flow for Your App
The critical distinction the previous sessions got wrong
There are two different flows in @react-oauth/google that return completely different things:

Component/Hook	Returns	Backend verification
<GoogleLogin /> component	credential (an ID token — a signed JWT)	Verify with google-auth-library → verifyIdToken()
useGoogleLogin() hook	access_token (an OAuth access token)	Fetch userinfo from Google API
The GoogleLogin component returns { credential: '...', clientId: '...', select_by: '...' } while useGoogleLogin returns { access_token: '...', expires_in: 3599, scope: '...', token_type: 'Bearer' } 
GitHub
 — they are completely different objects requiring different backend handling.

Google's official guidance is explicit: do not accept plain user IDs on your backend server — always use verifiable ID tokens to securely identify users. 
Google

The Two Valid Approaches
Option A — <GoogleLogin> component + ID Token (Google's recommended approach)

This is the correct, secure flow for sign-up/sign-in only (which is your use case):

1. User clicks button → GoogleLogin renders Google's button
2. User selects account → Google returns credential (ID token JWT)
3. Frontend sends credential to your backend POST /api/auth/google
4. Backend calls verifyIdToken() from google-auth-library
5. Backend extracts sub (unique permanent Google ID), email, name
6. Backend finds or creates user in DB
7. Backend returns your app's JWT
8. Frontend stores JWT → navigate to /compliance
The sub claim from the ID token payload is the unique, permanent identifier for the Google account — it's the right primary key to store, not the email (which users can change). 
Google

Option B — useGoogleLogin() hook + access token → userinfo fetch

1. User clicks custom button → useGoogleLogin() opens popup
2. Google returns access_token
3. Frontend sends access_token to backend
4. Backend fetches https://www.googleapis.com/oauth2/v1/userinfo?access_token=...
5. Backend gets email, given_name, family_name, id (Google's user ID)
6. Backend finds or creates user, returns your app's JWT
This works but is less secure — the access token is short-lived, can't be cryptographically verified, and Google notes if your app only uses email, profile, and openid scopes for authentication, using a JWT ID token is the most straightforward and simple method available. 
Google

Which to use for AMAX-FX
Use Option A (<GoogleLogin> + ID token). Reasons:

You want a custom-styled button — <GoogleLogin> supports useOneTap and custom render but for a fully custom button, use GoogleLogin with render prop or switch to useGoogleLogin with flow: 'auth-code'
Your backend already has google-auth-library mentioned — use verifyIdToken(), not a userinfo fetch
Google's best practice guidance says to use the sub claim as the unique and permanent user identifier in your system

  });
