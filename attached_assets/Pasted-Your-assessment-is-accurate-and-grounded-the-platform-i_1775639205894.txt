## What “real backend internal transfer” should include

At minimum, the implementation should:

### 1. User resolution

* Input: email (or phone if you extend later)
* Resolve:

  * `email → userId`
  * Validate user exists and is active

---

### 2. Balance validation

* Check sender wallet balance (per asset)
* Enforce:

  * sufficient funds
  * optional limits (per transaction / daily)

---

### 3. Atomic ledger movement

You need a **double-entry style ledger**, e.g.:

For a transfer of 1.0 BTC:

* Debit sender wallet: -1.0 BTC
* Credit recipient wallet: +1.0 BTC

This must be:

* Atomic (transactional)
* Idempotent (prevent double execution)
* Auditable (immutable transaction record)

---

### 4. Transaction record

Store:

* senderUserId
* recipientUserId
* asset (BTC, ETH, etc.)
* amount
* timestamp
* status (pending → completed → failed)
* reference ID

---

### 5. Concurrency & safety

* Lock wallet balance during transfer (or use row-level DB transactions)
* Prevent race conditions
* Prevent negative balances

---

### 6. Notification layer (optional but expected)

* Email notification to recipient
* Activity log for both users

---

## Suggested API shape

A typical endpoint:

```
POST /api/wallet/transfer/internal
```

Payload:

```
{
  "asset": "BTC",
  "amount": 0.5,
  "recipientEmail": "user@example.com"
}
```

Backend flow:

1. Authenticate sender
2. Resolve recipient by email
3. Validate asset + balance
4. Execute atomic ledger transaction
5. Return transaction ID

---

## Architecture note (important)

You should treat this as part of a **core ledger service**, not just a wallet helper.

Ideal separation:

* Wallet UI → initiates transfer
* Wallet service → orchestrates
* Ledger service → executes accounting entries

---

## Edge cases to handle early

* Sending to self
* Invalid / unregistered email
* Partial failures (ensure rollback)
* Decimal precision (especially for crypto assets)
* Idempotency keys (to prevent duplicate submissions)
